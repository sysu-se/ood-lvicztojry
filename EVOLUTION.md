## 二、提示（Hint）功能实现

### 2.1 提示功能归属分析

经过深入思考，我将提示功能的实现分配如下：

**主要归属于 Sudoku 对象**：

- `getCandidates(row, col)`：计算指定位置的候选数集合
- `getNextHint()`：计算下一个推荐填写的位置和数字
- `canContinue()`：检查是否还有可填入的位置

**Game 对象协调**：

- 提供对外统一的提示接口
- 处理提示与游戏状态的交互

**设计理由**：

- 候选数计算依赖于当前棋盘状态（行、列、宫的数字分布），这是Sudoku的核心职责
- 数据局部性原则：提示算法需要频繁访问grid数据，放在Sudoku中更合理
- 职责内聚：Sudoku对象专注于棋盘状态相关操作

### 2.2 候选提示功能

#### 2.2.1 实现原理

```javascript
getCandidates(row, col) {
  // 1. 检查该位置是否已有数字
  // 2. 遍历1-9数字，检查是否违反数独规则
  // 3. 返回所有合法数字的集合
}
```

#### 2.2.2 算法细节

- **行检查**：遍历当前行，排除已存在的数字
- **列检查**：遍历当前列，排除已存在的数字
- **宫检查**：遍历3x3宫格，排除已存在的数字
- **时间复杂度**：O(1)，因为只检查固定数量的单元格

#### 2.2.3 边界条件处理

- 位置超出边界时抛出错误
- 位置已有数字时返回空数组
- 输入验证确保参数合法性

### 2.3 下一步提示功能

#### 2.3.1 实现策略

```javascript
getNextHint() {
  // 1. 优先查找唯一候选数的位置（确定解）
  // 2. 如无确定解，返回第一个有候选的位置
  // 3. 如无法继续，返回null
}
```

#### 2.3.2 智能提示算法

- **确定解优先**：首先寻找只有一个候选数的位置
- **启发式搜索**：在多个候选位置中选择最优的
- **可继续性检查**：确保提示不会导致死局

## 三、探索（Explore）模式实现

### 3.1 探索模式的本质

探索模式是一种**状态分支机制**，允许用户在不确定如何继续时进行尝试，然后决定是否保留这些尝试。

### 3.2 设计方案选择

对比多种实现方案：

| 方案         | 优点                     | 缺点                       | 选择结果 |
| ------------ | ------------------------ | -------------------------- | -------- |
| 独立探索对象 | 职责清晰，易于扩展       | 增加复杂性，状态同步困难   | ❌       |
| Game状态切换 | 简单直观，与现有逻辑一致 | 增加Game对象复杂度         | ✅       |
| 临时快照机制 | 实现简单                 | 功能受限，难以支持复杂操作 | ❌       |

**最终选择**：Game状态切换方案，因为它与现有架构最兼容。

### 3.3 探索模式状态管理

#### 3.3.1 核心状态变量

```javascript
// Game对象中的探索相关状态
exploring: boolean,           // 当前是否处于探索模式
explorationStartSudoku: Sudoku, // 探索开始时的棋盘状态
explorationMoves: Move[],     // 探索期间的操作记录
```

#### 3.3.2 主局面与探索局面关系

**复制关系**：

- 探索开始时，克隆当前Sudoku状态作为探索起点
- 探索期间的所有操作仅影响克隆对象
- 保持主局面的完整性

**独立操作**：

- 探索期间的修改不影响主局面
- 拥有独立的历史记录系统
- 支持探索内的撤销操作

**延迟合并**：

- 探索结果可以选择性地合并到主历史
- 支持提交或放弃探索结果

### 3.4 探索模式操作流程

#### 3.4.1 进入探索模式

```
1. 保存当前Sudoku状态到explorationStartSudoku
2. 重置explorationMoves为空数组
3. 设置exploring标志为true
4. 通知UI更新状态
```

#### 3.4.2 探索期间操作

```
1. 所有操作记录到explorationMoves而非主历史
2. 操作影响explorationStartSudoku的克隆副本
3. 支持在探索模式内的撤销操作
```

#### 3.4.3 提交探索结果

```
1. 将explorationMoves合并到主历史
2. 清空探索相关状态
3. 设置exploring标志为false
4. 使主历史中的redo栈失效
```

#### 3.4.4 退出探索模式

```
1. 恢复到explorationStartSudoku状态
2. 清空探索相关状态
3. 设置exploring标志为false
4. 通知UI状态更新
```

### 3.5 探索模式约束

#### 3.5.1 操作限制

- **重做限制**：探索模式下不支持重做功能
- **冲突检测**：提供isConflicted()方法检测探索局面是否无效
- **状态检查**：确保只能在合适的状态下执行相应操作

#### 3.5.2 用户体验考虑

- 探索模式下UI会有明显的视觉反馈
- 提供清晰的提交/退出选项
- 防止意外的数据丢失

## 四、历史（History）演进分析

### 4.1 历史结构变化

**Homework 1 结构**：

```
主历史: [move1, move2, move3, ...] ← 当前状态
```

**Homework 2 结构**：

```
普通模式:
主历史: [move1, move2, move3, ...] ← 当前状态

探索模式:
主历史: [move1, move2, move3] ← 探索起点
探索历史: [exp_move1, exp_move2, ...] ← 当前状态
```

### 4.2 撤销/重做行为变化

#### 4.2.1 普通模式行为

- `undo()`：从主历史栈弹出操作
- `redo()`：从redo栈弹出操作并重新执行

#### 4.2.2 探索模式行为

- `undo()`：从探索历史栈弹出操作，直到探索历史为空时退出探索
- `redo()`：在探索模式下被禁用（避免复杂状态管理）
- `exitExploration()`：恢复到探索前状态
- `submitExploration()`：将探索历史合并到主历史

### 4.3 历史一致性保证

#### 4.3.1 事务性操作

- 探索提交是原子操作，要么全部提交，要么全部回滚
- 探索退出完全恢复到探索前状态

#### 4.3.2 状态完整性

- 无论何种操作，都保证Sudoku状态的有效性
- 历史记录始终保持一致性

## 六、Homework 1 设计局限性分析

### 6.1 状态分支支持不足

**问题**：HW1的线性历史模型无法自然支持分支和回溯需求
**解决方案**：引入双历史模型（主历史+探索历史）

### 6.2 高级策略支持缺失

**问题**：原始设计只考虑基础操作，未考虑提示和探索等高级功能
**解决方案**：扩展Sudoku对象的智能分析能力

### 6.3 职责边界灵活性

**问题**：某些边界在面对新需求时显得僵硬
**解决方案**：在保持核心职责的同时，适当扩展对象能力

## 八、关键技术决策

### 8.1 深拷贝策略

- **目的**：确保探索操作不会意外影响主局面
- **实现**：使用Sudoku.clone()创建棋盘的完整副本
- **时机**：进入探索模式和提交探索时

### 8.2 历史合并策略

- **提交时**：直接将探索期间的移动追加到主历史末尾
- **保持连续性**：确保历史的逻辑连续性
- **副作用处理**：提交后使redo栈失效

### 8.3 状态一致性

- **统一通知机制**：确保UI在状态变化时得到及时更新
- **事务性操作**：确保复合操作的原子性
